White Box Testing

What is White-box Testing

When we test the internal structure of a component or system by looking at the implementation, then this method of testing is known as white-box testing. But let's understand why is this called white box testing. This is because the item under test is like a white or transparent box to us, inside which the tester can clearly see. The tester provides inputs , exercise paths through the code and check the appropriate outputs.

White-box testing is also known as Structure-based testing, glass box or transparent box testing

Test levels applicable

White-box testing method can be applied to Component testing, Integration testing and system testing. Some examples of white-box testing at different levels are as follows:

1. Component Testing : Software component's structure like statements, decisions, paths or branches.

2. Integration Testing : Call tree structure

3. System Testing : Any structure like web page structure, menu structure or business process.

White Box test design techniques

White-box test design techniques are the procedure used to derive test cases on the basis of analysis of the internal structure of a component or a system.

Different white-box test design techniques are as follows.

1. Statement Testing and Coverage

Statement coverage determines the percentage of executable statements that are exercised by a test case suite. It is calculated as Number of statements exercised/Total number of statements. Execution is done in a way that at least each statement in the code is executed once . This is normally measured with the help of a software tool.

2. Decision Testing and coverage

Decision coverage determines the percentage of decision outcomes that are exercised by a test case suite. It is calculated as Number of decision outcomes covered by test cases/Total number of possible decision outcomes in the code. Branch testing is related to this coverage.Decision testing is a kind of control flow testing as a particular flow of control through the decision points is tested. Decision coverage provides better coverage than statement coverage.

3. Condition Coverage

Condition coverage determines the percentage of condition outcomes that are exercised by a test case suite. A condition is a logical expression which can be evaluated as True or False.When each of the condition in a decision statement is set to true and false once then it leads to 100% condition coverage.

4. Modified condition decision coverage

Modified condition decision coverage determines the percentage of all conditions which independently affect the decision result that are exercised by a test case suite. All the conditions are tested for True and False value and each should affect the decision outcome independently. It is basically used in avionics testing as per DO178B. It is also known as Modified multiple condition coverage

Advantages of White Box testing

1. This is a good candidate for test automation.

2. Testing can be started at early stage without GUI even being complete.

3. More thorough testing can be achieved as we cover most code paths.

Disadvantages of White Box testing

1. Highly skilled resources with good programming knowledge are required since these tests are complex tests

2. Tools availability for each type of application can be difficult.

3. The main focus remains on the software and missing functionality may not be uncovered as a result.
