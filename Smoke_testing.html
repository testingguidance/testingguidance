Smoke and Sanity Testing

What is Smoke Testing

Whenever a new software build is released from development team, the first step is to check the stability of the build i.e. if the build is stable enough that further testing is done on the build. It may be the case that the new build has installation defects, show stopper defects or the most critical functionality is broken. The testing performed to cover these aspects is known as Smoke testing . The purpose of this testing is to decide if the new build needs to be accepted or rejected based on smoke test results.Thus this is also known as Build Verification Testing. The interesting name "Smoke testing" in software has been derived from hardware testing where the device is tested for the first time to check whether it catches fire or smoke the first time it is used. Once the smoke tests pass, further functional or regression tests are planned as smoke testing is done on initial builds or enhancement builds.

Example of smoke tests for web applications can include tests to validate the website is launched successfully, the login/logout tests, 1-2 end to end tests etc.

What does smoke testing cover?

1. Smoke testing test suite has most critical functional tests of the software/application considering mainly the positive paths.

2. Smoke tests are non-exhaustive set of tests and don't cover functional tests in depth.

3. Smoke tests can also cover the tests for installation or configurations for desktop applications (if applicable).

4. Smoke tests are normally done at Integration testing, System testing and acceptance testing test levels.

Who performs Smoke testing?

Both developers or testers can perform smoke testing. In our experience with different projects, either of the teams used to perform smoke testing. Normally when builds are released on dev environments or shared environments between development and testing teams then developers perform quick smoke tests before passing it to testing team. When there is a build deployed on testing environment then normally testers perform it. In agile projects, either of the team members can perform as per availability.

Automation and Continuous Integration

Smoke tests are good candidate for automation for stable applications. With each build, the same tests need to be executed which might take longer time when performed manually. So it is always better to automate the smoke tests by choosing an appropriate tool for automation. Furthermore, the next effective step is to integrate the automated smoke tests into a continuous integration tool (CI tool). CI tools will automatically start the smoke tests when a new build becomes available , run tests and pass the results .

Advantages of Smoke testing

1. When smoke tests pass, it gives confidence in the build and further testing can be carried.

2. When critical smoke tests fail, then the build is rejected and saves testing effort and time.

3. Smoke testing helps in finding integration and major problems early in the development cycle when there are many components or sub systems in the application.

4. In software development life cycles where there are regular build for example in agile projects, smoke testing becomes really important to adhere to project timelines.

Another important topic which is often used along with Smoke testing is Sanity Testing. These terms are not identical or synonym of each other. However , because of some commonalities between these two types of testing, they are referred together. we are also covering them together so that their similarities or rather their dissimilarities become clearer.

What is Sanity Testing

When we perform testing on a build which contains minor changes or defect fixes to verify these changes along with most critical functional tests, then this is known as Sanity testing. Normally this testing is performed near the release of the software when we get final builds after defects found in the regression cycle are fixed. At that time, there is no point of running the entire regression suite again as code changes are normally controlled and we want to make sure that the new build has not introduced any show stopper defect and critical functionality is intact, so sanity testing comes into the picture. Also, when we have patch releases i.e small releases with only bug fixes and no enhancement features, then the scope of testing becomes sanity testing along with bug verification. The name derives from the fact that tester checks if the software has been delivered with "sanity" in it.

Example of sanity tests for banking applications like mortgage application is to create a new loan application with the mandatory fields.

What does sanity testing cover?

1. Sanity tests cover tests from regression test suite majorly around the changed areas and bug fixes.

2. Sanity tests are sub set of regression test suite and cover the highest priority regression tests .

Who performs Sanity testing?

Normally testing team performs sanity testing.

Advantages of Sanity testing

1. When sanity tests pass, it gives confidence in the build to be released.

2. For builds with minor changes or only having defect fixes, sanity tests help in completing the testing cycles faster by running highest priority regression tests.
