Black Box Testing

What is Black Box Testing

When we test a component/software system/application to ascertain if it produces the expected outcome without looking at the internal code implementation, then this method of testing is known as black box testing. But let's understand why is this called black box testing. This is because the item under test is like a black box to us, we can't really see what is inside it. We just know that given x inputs , the system should provide us y outputs. This testing method is both functional and non functional. At one hand, we can check for the expected functional behaviour and on the other hand we can also test if non-functional characteristics like performance etc is as per requirements.

Black box testing is also known as Specification-based testing, opaque-boxed or closed-box testing

Example to understand Black box testing.

Imagine a new android mobile phone is launched in the market and as a tester you have been asked to test it. Few of the specifications are as below:

-> The user is able to make calls . (Functional)

-> The user is able to click pictures through camera (Functional) etc ..

-> The mobile phone doesn't get heated after being continuously used for 4 hours (Non functional)

The testing of all above features doesn't require any knowledge of the code and is hence black box testing

Test levels applicable

Black box testing method can be applied to all levels of software testing : Component testing, Integration testing, system testing and acceptance testing.

Black Box test design techniques

Black box test design techniques are the procedure used to derive test cases on the basis of functional or non-functional specifications of a component or system without referencing the internal structure or code.

Different black box test design techniques are as follows. Their detailed description can be found by clicking on the names below

1. Equivalence partitioning

The inputs to the system or software that have the same expected behaviour are grouped together. This is done so that they are used in the same way. These are known as equivalence partitions or classes.

Let's take an example, in an online application form for club membership, the user needs to enter his/her age. The club membership can only be taken if you are 18 or above till 65 years only.The range of the age is to be considered between 0to 100. Now let's go through some related points below.

1.Equivalence classes are created for valid data i.e the data that would accepted.

The valid equivalence class would be [18,65] inclusive.

2.Equivalence classes are created for invalid data i.e. the data which will be rejected.

The invalid equivalence classes would be [0,18[ (0 to less than 18),]65,100]( greater than 65 to 100)

3.Equivalence classes can be created for output values, internal values and interface parameters also.

4. Equivalence classes can be applied at all test levels.

2. Boundary value analysis

Boundaries at equivalence partitions and other test data are critical areas to be tested as there are good chances of catching defects there. The maximum and minimum values of equivalence partition are the boundary values. This technique normally supplements equivalence partitioning and other black box test design techniques.

In the example above given for equivalence partitioning ,the boundary values 18 and 65 are the boundary values which needs to be tested. It's a common error in the code that >, <, == are used incorrectly which can be caught using these boundary values.

3. Decision table

When the system requirements contain logical conditions or complex business rules , decision tables are effective way of testing such conditions. This is also known as Cause effect table.The related important points are below

1. To create decision table, analyse the specifications.

2. The inputs in the form of conditions and actions are normally specified as boolean i.e. can be true or false.

3. The decision table will contain input conditions and associated actions .

4. Each column of the decision table contains a business rule with combinations of input variables which would result into actions associated with that business rule.

5. All the columns in the decision table need to be considered as test cases to ensure full coverage

Let's extend our example taken under Equivalence partitioning. The rules goes like this:

Specification says : No discounts can be given unless discount eligibility is met. After the eligibility criteria is met, provide 20% discount for army personnel and 25% for disadvantaged category

There will be 4 rules. The first condition (Is the eligibility criteria met?) has two possible outcomes, yes or no. The second condition (type of discount) has two possible outcomes, Army Personnel (A) or Disadvantage Category (D) Two times two is four. These are the four test cases to be considered. 4. State-transition testing

This testing is primarily used for systems which may exhibit different behaviour based on current or previous states. State transition diagrams help in understanding and testing these systems. The main items are as follows:

1. States of the software

2. Transitions between states

3. The events or inputs which trigger these transitions or state changes

4. Actions resulting from these transitions

State table shows these components and also invalid transitions. The idea is to test all states, all transitions along with invalid transitions. State transition testing is used normally for embedded software and technical automation. Let us take an example from the figure below.

5. Use Case Testing

Use case testing is used to derive test cases from use cases to uncover defects in the process flows which might occur during the real world use of the system. Related points are mentioned below

1. Users or systems in use cases are termed as actors

2. The interactions between these actors produce results to the customer or system user

3. Each use case has preconditions (necessarily to be met before) and post conditions with results

4. The use cases have main and alternative scenarios.

Advantages of Black Box testing

1. Test activities can be started after the specifications are complete.

2. As this method is specification based, discrepancies can be found in specifications.

3. The testing team performs this testing independently without having any bias towards implementation thus achieving testing objectives efficiently.

4. Testers with limited programing knowledge can also perform this testing.

5. As this is applied at integration testing level, communication between components or systems can be validated.

Disadvantages of Black Box testing

1. There are always chances of program paths being missed out. The code can contain dead code, unreachable code or redundant code which will not be uncovered using this method.

2. This method requires specifications to be written completely . If this is not the case, test case designing may be difficult and can lead to incomplete testing.
